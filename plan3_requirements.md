# 要件定義書 `plan3`

## プロダクト名
**ZEN Inbox**

## 1. 背景
Gmail、Slack、LINE などに予定・返信依頼・締切・契約更新情報が分散しており、必要な情報を毎回探すのが面倒である。  
既存の GPTs でも相談や要約はできるが、毎回ユーザーが情報を持ち込む必要があり、継続的な整理状態を持ちにくい。

そのため本企画では、`AIチャット` ではなく、**接続済み情報を継続的に整理し、行動可能な状態に保つ個人用オペレーション画面** を作る。

## 2. テーマ解釈
本企画における `ZEN` の定義は以下とする。

**ZEN = ノイズを減らし、今やるべきことを静かに見える化すること**

精神論ではなく、情報過多な日常に対して、認知負荷を下げて行動しやすくする UX として扱う。

## 3. プロダクト概要
`ZEN Inbox` は、外部サービスに散在する連絡や通知を収集し、AI とルールベース分類によって整理し、ユーザーにとって重要なものを一覧で提示するダッシュボードである。

v1 では、**Gmail を主対象とした一覧中心の整理画面** を作る。  
チャットアプリやメールクライアントを置き換えるのではなく、既存の運用の上に載る `整理レイヤー` として設計する。

主目的は `相談` ではなく、以下の状態を作ること。

- 探さなくてよい
- 優先度が見える
- 未処理が残らない
- 次に取る行動が分かる

## 4. GPTsとの違い
本プロダクトは GPTs と以下の点で明確に異なる。

### 4.1 情報を毎回貼らなくてよい
- GPTs は都度ユーザー入力が必要
- ZEN Inbox は接続済みデータを前提に、最初から整理された状態を見せる

### 4.2 継続状態を持つ
- GPTs は単発の会話に強い
- ZEN Inbox は `未処理 / 保留 / 今日やる / 要返信` などの状態を持続管理する

### 4.3 実務整理に近い
- GPTs は提案までが中心
- ZEN Inbox はラベル付けや状態更新など、実際の整理行動を前に進める

### 4.4 既存メールクライアントとの違い
- Notion Mail などの新しいメールクライアントに乗り換える前提ではない
- ZEN Inbox は Gmail をそのまま使い続けながら、`今やるべきもの` の把握に特化する
- v1 は `多機能なメール管理` ではなく `静かな実務整理` を優先する

## 5. ターゲットユーザー
### メインターゲット
- Gmail を日常的に使う会社員・知的労働者
- メールや通知の中から予定や返信事項を探している人
- 高機能ツールより、まず「今どうなっているか」を一目で把握したい人

### サブターゲット
- 就活生、学習者、フリーランス
- 契約更新や請求通知を見落としやすい人
- 将来的に Slack / LINE / Gmail を横断して整理したい人

## 6. 解決する課題
- 予定や締切が複数サービスに分散している
- 返信が必要な連絡が埋もれる
- 請求や契約更新が放置される
- 情報はあるが、次に何をすべきかが分からない
- 毎回検索・確認・再整理するのが面倒

## 7. 提供価値
- 散在した連絡を一箇所で見ることができる
- AI が重要度と種別を整理する
- ユーザーは `判断` に集中できる
- 整理状態が継続し、日々の確認コストが下がる

一言で言えば、

**散らばった連絡を、行動できる状態に整えておく個人用整理画面**

## 8. v1 の方針
### 8.1 中核価値
v1 の中心価値は以下とする。

**探す手間をなくし、今やるべきものを一覧で見える化すること**

加えて、v1 の設計原則を以下とする。

- 主役は一覧ダッシュボードであり、チャットは補助とする
- AI が使えない環境でも最低限成立する
- 外部連携が失敗しても UI 全体は崩れない
- 既存メール運用を置き換えず、上に被せる整理体験を目指す

### 8.2 連携対象
- v1 は `Gmail先行`
- Slack / LINE は実連携しない
- ただし将来拡張できる設計にする
- Gmail 実連携は行うが、UI と分類基盤は Gmail 固有仕様に依存しすぎない
- 開発中およびデモ時は `モックデータ / 手入力データ / Gmailデータ` を切り替え可能にする

### 8.3 自動化レベル
- v1 は低リスク操作のみ
- Gmail への書き戻しは `ラベル付けのみ`
- 削除、解約送信、自動返信は行わない

### 8.4 AI 実装方針
v1 の AI は単一チャットではなく、役割分担型の整理パイプラインとして扱う。

- `kind classifier`
  返信要否、予定、締切、契約系などの分類
- `priority judge`
  `urgent / soon / someday` の優先度判定
- `action suggester`
  次の1アクションの生成
- `noise filter`
  メルマガ、重複、低優先ノイズの抑制

上記は AI で強化するが、各段階はルールベースでも代替可能にする。  
ハッカソンの実装では、`マルチエージェント風の役割分担` を見せつつ、実装は軽量なモジュール分離で成立させる。

## 9. 情報の分類
v1 では取得情報を以下の5種に分類する。

- `reply_needed`
  返信が必要
- `scheduled`
  日時や予定を含む
- `deadline`
  締切や期限がある
- `read_later`
  後で確認すべき
- `subscription_candidate`
  請求、契約更新、無料期間終了など

さらに、優先度として以下の3区分を持つ。

- `urgent`
- `soon`
- `someday`

## 10. UI 要件
### 10.1 主画面
主画面は `整理ダッシュボード` とする。  
チャットではなく、一覧を主役にする。

UI のトーンは、`静かに整理できること` を優先する。  
過度な通知感や煽る表現を避け、短時間で優先事項を判断できる見た目にする。

### 10.2 表示要素
各アイテムには最低限以下を表示する。

- 件名または要約
- 送信元
- 受信日時
- 分類種別
- 優先度
- 推定アクション
- 元メールを開く導線
- ラベル付け操作

### 10.3 画面構成
- 上部: 今日見るべき件数、未処理件数、返信必要件数などのサマリー
- 中央: `urgent / soon / someday` の一覧
- 補助領域: チャットによる絞り込み・再整理
- 補助領域: 接続状態、同期状態、ラベル操作結果

### 10.4 チャットの位置づけ
チャットは主機能ではなく、以下の補助用途に限定する。

- 一覧の再絞り込み
- 特定条件の検索
- 要約の再生成
- 今週やるべきことの再提示

チャット応答の文体は、必要に応じて `ZENらしい落ち着いた案内` を採用してよい。  
ただし演出が主目的ではなく、判断のしやすさを優先する。

## 11. 機能要件
### 11.1 データ取得
- Gmail から対象メールを取得できる
- 対象メールは受信日時、件名、本文、差出人、スニペットを保持する
- 初期表示用に直近メールや未読メールを対象にできる
- Gmail 取得に失敗した場合でも、アプリは `空状態 / エラー状態 / デモデータ表示` のいずれかに安全に遷移できる
- 将来の Slack / LINE 対応を想定し、取得データは共通の中間表現へ正規化する

### 11.2 整理処理
- 各メールを 5種の分類に振り分ける
- 優先度を `urgent / soon / someday` に振り分ける
- 推定アクションを生成する
- 重複や明らかなノイズを抑える
- 各整理処理は独立して失敗してもよく、可能な範囲で部分結果を返せる
- AI が使えない場合はルールベース分類へ自動フォールバックする
- 必要に応じて、どの整理処理が判定したかを UI 上で簡潔に示せる

### 11.3 一覧表示
- 分類済みアイテムを一覧で表示できる
- 優先度別にフィルタできる
- 種別別にフィルタできる
- 元メールを開ける
- 初期表示だけで `今日見るべき件数 / 未処理件数 / 返信必要件数` を把握できる
- 1件ごとに `件名または要約 / 送信元 / 受信日時 / 分類 / 優先度 / 推定アクション` を確認できる

### 11.4 Gmail 書き戻し
- アイテムに対して Gmail ラベルを付与できる
- v1 の対象ラベル例:
  - `ZEN/Urgent`
  - `ZEN/Soon`
  - `ZEN/Someday`
  - `ZEN/Reply`

### 11.5 補助チャット
- 一覧データを前提に自然文で再整理できる
- 例:
  - 「返信が必要なものだけ」
  - 「今週中に片付けるべきもの」
  - 「契約系だけ見せて」

## 12. 非機能要件
- Google Chrome で動作する Web アプリであること
- 一覧中心で短時間に状況把握できること
- AI 未利用時でも最低限のルールベース整理が動くこと
- 外部連携失敗時でも UI が壊れないこと
- 将来の Slack / LINE 追加に耐える構造であること
- Chrome Built-in AI が使える環境では、オンデバイス AI を優先利用できること
- オンデバイス AI が使えない場合でも、アプリの中心価値が失われないこと
- デモ時に Gmail 実データが使えない場合でも、モックデータで体験を再現できること
- 判定理由や状態表示は、過剰な専門用語なしに理解できること

## 13. 連携ロードマップ
### v1
- Gmail 実連携
- 一覧ダッシュボード
- ラベル付け
- 補助チャット
- 役割分担型の整理パイプライン
- ルールベースフォールバック
- モックデータ切り替え

### v1.5
- 手入力 / コピペ / 転送メール取り込み
- Gmail 以外の情報を仮統合

### v2
- Slack 実連携
- LINE 実連携
- ソース横断の統一 Inbox

### v3
- サブスク整理の強化
- 解約候補の精度向上
- 解約支援フローの追加検討

## 14. データモデル方針
### 主な型
- `InboxSourceAdapter`
- `RawSourceItem`
- `InboxItem`
- `InboxKind`
- `InboxBucket`
- `InboxState`
- `InboxChatRequest`
- `InboxChatResponse`

### source の想定値
- `gmail`
- `manual_import`
- `slack`
- `line`

## 15. 安全性ポリシー
- メール削除はしない
- 自動返信しない
- 解約を勝手に進めない
- 外部サービスへの書き戻しはラベル付けに限定
- AI の判定は補助であり、最終判断はユーザーが行う
- AI の各整理モジュールが不確実な場合は、断定ではなく候補として表示する
- プライバシー上の観点から、利用可能であればオンデバイス AI を優先する

## 16. 成功条件
この企画が成功といえる条件は以下。

- Gmail 内の必要情報を探す回数が減る
- 一覧を見るだけで優先事項が分かる
- 返信漏れや締切見落としが減る
- Gmail 側の整理状態も改善する
- ユーザーが「これはGPT相談ではなく、整理作業の実務ツールだ」と感じる
- 審査やデモにおいて、`一覧整理 + AI役割分担 + 安全なフォールバック` が一貫して伝わる

## 17. 非目標
v1 では以下を対象外とする。

- Slack / LINE の実連携
- カレンダー完全同期
- 自動解約
- 自動送信
- 自動削除
- 全メールの完全自動分類
- メールクライアントの完全置き換え
- Notion 連携やタスクDB化の深掘り
- 複雑なエージェント会話UI

## 18. 要約
`plan3` における `ZEN Inbox` は、AIチャットを作る企画ではない。  
Gmail などに散在する連絡を継続的に整理し、ユーザーが今やるべきことを一覧で把握できるようにする、個人用オペレーションダッシュボードである。  
v1 は Gmail 先行で、整理・分類・ラベル付けを中心に実装し、将来的に Slack / LINE へ拡張する。  
また、AI は単一チャットではなく `kind / priority / action / noise` の役割に分けた整理パイプラインとして扱い、AI が使えない場面でもルールベースで最低限成立する設計とする。
